# LangGraph Persistent RAG Architecture Design

## Goal

Make the current LangGraph analysis workflow the only application architecture.
Migrate authentication, user isolation, durable file storage, OCR, and PostgreSQL
persistence from the legacy routes into that architecture. Keep RAG and Milvus as
required parts of document parsing and candidate analysis.

After migration, uploaded data, parsed profiles, evidence, match reports, and
question sets must survive frontend refreshes and backend restarts.

## Scope

### Keep And Formalize

- The LangGraph candidate workflow:
  `load_documents -> index_documents -> parse_jd -> parse_resume ->
  retrieve_evidence -> evaluate_match -> calculate_score ->
  generate_questions -> persist_report`.
- Evidence-backed matching with Milvus retrieval.
- Structured LLM output validated by Pydantic schemas.
- Python-calculated weighted total scores.
- The current Vue document, matching, question, and follow-up workflows.

### Migrate From The Legacy Architecture

- Bearer-token authentication and per-user data isolation.
- PostgreSQL document and analysis persistence.
- Uploaded-file storage.
- PDF text extraction and scanned-PDF OCR.
- Resume and JD deletion with related vector cleanup.

### Remove After Migration

- Legacy `/resume/*`, `/jd/*`, and `/analysis/*` routes.
- The in-memory run repository.
- Duplicate legacy parsing, matching, embedding, and vector-store paths.
- All `demo_*` module and class names.
- Frontend behavior that stores original files in memory and re-uploads them for
  matching.

Chat and authentication routes remain outside this removal scope.

## Architecture

### PostgreSQL Responsibilities

PostgreSQL is the source of truth for:

- Users and ownership.
- Uploaded document metadata, stored-file path, raw text, structured profile,
  parse status, and timestamps.
- Analysis runs and their selected JD and resumes.
- Candidate reports, criterion evaluations, scores, recommendations, and
  summaries.
- Generated formal and follow-up question sets.

Every document, run, report, and question query must be filtered by the current
authenticated user.

### File Storage Responsibilities

Uploaded files are stored under the application data directory using safe,
deduplicated filenames. The database stores the resulting path. Deleting a
document removes the database record, stored file, and associated Milvus data.

### Milvus Responsibilities

Milvus is required for:

- JD and resume document chunks.
- Per-criterion resume evidence retrieval.
- Persisted analysis artifacts that can be searched later.

Chunk and artifact records include stable PostgreSQL IDs and `user_id` metadata.
All searches filter by `user_id`, run/document IDs, candidate ID, and document
type as applicable.

The application uses one embedding service and one Milvus implementation.
Embedding dimension is derived from the produced vector instead of requiring a
separate manually synchronized dimension configuration.

### LangGraph Responsibilities

LangGraph remains the orchestration layer for candidate analysis. Its state uses
durable database identifiers and loaded document chunks instead of uploaded file
objects.

The persistence node writes reports and question sets to PostgreSQL. Milvus
stores searchable evidence chunks and analysis artifacts, but is not the source
of truth for reports.

## Data Model

Use the existing document and analysis entities as a starting point, adjusting
them to match the LangGraph report schema:

- `JobDescription`: owner, filename or title, stored path when applicable, raw
  text, structured `JobProfile`, parse status.
- `Resume`: owner, filename, stored path, raw text, structured `ResumeProfile`,
  parse status.
- `AnalysisJob`: owner, JD reference, status, summary, timestamps.
- `AnalysisCandidate`: analysis job and resume reference, processing status.
- `MatchResult`: durable candidate report content, total score,
  recommendation, criterion evaluations, strengths, gaps, and summary.
- `InterviewQuestionSet`: formal questions generated by the graph.
- `FollowUpQuestionSet`: ambiguity follow-ups generated by the graph.

Schema changes must preserve existing local development databases through
additive initialization or an explicit migration strategy. No response may be
assembled from preset sample data.

## API Design

All application APIs require authentication except authentication endpoints.

### Documents

- `POST /api/documents`: upload one or more JD or resume files, persist files,
  extract text/OCR, parse profiles, chunk documents, and index chunks.
- `GET /api/documents`: list current user's persisted documents.
- `GET /api/documents/{document_id}`: return persisted document details.
- `POST /api/documents/{document_id}/parse`: reparse the stored file.
- `DELETE /api/documents/{document_id}`: delete the document and related vector
  data.

The document identifier must be a durable database-backed identifier, not a
random frontend-session identifier.

### Runs And Reports

- `POST /api/runs`: accept a persisted JD ID and persisted resume IDs, create an
  analysis job, and execute the LangGraph workflow.
- `GET /api/runs`: list persisted runs for the current user.
- `GET /api/runs/{run_id}`: return a persisted run summary.
- `GET /api/runs/{run_id}/candidates/{candidate_id}`: return a persisted report.
- `DELETE /api/runs/{run_id}/candidates/{candidate_id}`: delete a persisted
  candidate result and related artifacts.
- Evidence search remains available and is filtered by authenticated ownership.

The existing `/api/runs/analyze` multipart upload route is replaced by the
ID-based run creation route.

## Upload And Parsing Flow

1. Validate ownership context, filename, extension, file count, and size.
2. Persist the uploaded file using a safe deduplicated name.
3. Extract normal text.
4. For scanned PDFs or insufficient extracted text, run the existing OCR
   service and retain page boundaries.
5. Reject the document with a clear error if usable text is still unavailable.
6. Create chunks with durable document and user identifiers.
7. Parse the structured profile through `AgentHarness`.
8. Save the document and profile in PostgreSQL.
9. Index chunks and a profile artifact in Milvus.
10. Return the persisted document record.

Failures must not leave orphaned files, incomplete database rows, or misleading
success responses.

## Matching Flow

1. Validate that the selected JD and resumes belong to the current user and
   parsed successfully.
2. Create a persisted analysis job and candidate rows.
3. Load document profiles and chunks by durable IDs.
4. Execute one LangGraph candidate workflow per resume.
5. Retrieve evidence from Milvus for every JD criterion.
6. Evaluate matches only from retrieved evidence.
7. Calculate the total score in Python.
8. Generate formal and ambiguity follow-up questions.
9. Persist the report and question sets transactionally.
10. Return the persisted run summary.

Candidate failures are persisted as failed statuses with useful error details.
They must not silently return sample or previous-run data.

## Frontend Changes

- Uploads call the new persisted document API for both JD and resumes.
- The document store loads persisted documents on application startup.
- Matching sends document IDs instead of original browser `File` objects.
- Match and question stores load persisted results after refresh.
- Delete and reparse operations call backend APIs.
- Existing page structure and interaction patterns remain unchanged unless an
  API-backed state requires a small adjustment.

## Naming And Removal

Rename active modules to production names, including:

- `schemas/demo.py` to an analysis/report schema module.
- `services/demo_documents.py` to the unified document service.
- `services/demo_analysis.py` to the analysis service.
- `graph/demo_workflow.py` to the candidate analysis workflow.
- `rag/demo_milvus.py` to the unified RAG store.
- `repositories/demo_runs.py` is removed and replaced by a PostgreSQL
  repository.

Remove legacy routes and implementations only after the new APIs and frontend
have equivalent tested behavior.

## Error Handling And Observability

- API errors distinguish validation, unsupported documents, OCR failures, model
  failures, vector database failures, and persistence failures.
- Log each boundary using user ID, durable document/run/candidate IDs, filename,
  stage, and elapsed time.
- Never log full resumes, credentials, or access tokens.
- Transactions roll back database changes on failure, and file/vector cleanup is
  attempted where needed.

## Testing

### Backend

- Uploading a resume persists the file, database row, parsed profile, and Milvus
  chunks derived from the uploaded content.
- Scanned PDF uploads use OCR.
- Upload failures do not leave orphaned state.
- Users cannot access each other's documents, runs, reports, or evidence.
- Run creation loads persisted document IDs and does not accept/rely on uploaded
  files.
- LangGraph retrieval filters evidence by user, run, candidate, and document.
- Reports and question sets remain available through a new repository/session.
- No endpoint returns preset sample data.
- Legacy routes return 404 after removal.

### Frontend

- Document upload sends files once and stores returned durable IDs.
- Refresh reloads persisted documents and match results.
- Matching sends JD and resume IDs without multipart file uploads.
- Reparse and delete operations update persisted backend state.

### Verification

- Run the backend test suite.
- Run the frontend test suite and production build.
- Exercise upload, refresh, match, report retrieval, evidence retrieval, and
  deletion against a local backend with PostgreSQL and Milvus.

## Implementation Order

1. Introduce production names and unified schemas/services without changing
   behavior.
2. Add durable document upload, list, detail, reparse, and delete APIs.
3. Replace the in-memory run repository with PostgreSQL persistence.
4. Adapt LangGraph and RAG filters to durable IDs and authenticated ownership.
5. Add ID-based run creation and persisted report APIs.
6. Migrate the frontend to persisted APIs.
7. Remove legacy routes, duplicate implementations, and remaining `demo_*`
   names.
8. Complete integration verification and cleanup.
